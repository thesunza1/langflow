Kế hoạch chi tiết: Chạy Flow Song Song

Cho phép nhiều luồng build/run đồng thời trên cùng một flow — dành cho các nhánh độc lập A→B và C→D (cập nhật sau review)

1. Mục tiêu & Phạm vi

Vấn đề hiện tại

Khi flow có 2 nhánh độc lập:

A (ChatInput)
→
B (Component)
Run B → gọi API build
C (TextInput)
→
D (Component)
Run C → bị chặn
Luồng thực tế sau multi-build khi click Run C khi B đang chạy:
1. handleClickRun → không kiểm tra isBuilding nữa
2. buildFlow tạo BuildInstance #2 → stopNodeId = C
3. Conflict check: C ≠ B → không conflict → chạy song song
4. Cả 2 SSE connections hoạt động độc lập

Phạm vi

  • Trong: Frontend store, build queue, event handling, giao diện per-build status
  • Ngoài: Graph engine core (lfx), caching strategy, component API

3. Giải pháp kiến trúc — Multi-build Instance Model

3.1 Data Model

// Mỗi lần click Run → một BuildInstance interface BuildInstance { buildId: string; // UUID stopNodeId: string; // Component được click startNodeId?: string; // (nếu có) status: 'pending' | 'running' | 'completed' | 'error' | 'cancelled'; // Per-instance state (thay thế global verticesBuild) verticesBuild: { verticesIds: string[]; verticesLayers: VertexLayerElementType[][]; verticesToRun: string[]; runId: string; // run_id từ backend } | null; // Per-instance build status (thay thế global flowBuildStatus) flowBuildStatus: Record<string, { status: string; timestamp?: string }>; // Per-instance flow pool (thay thế global flowPool) flowPool: Record<string, VertexBuildTypeAPI>; // Build controller để cancel riêng abortController: AbortController; // Callbacks vẫn giữ nguyên nhưng cập nhật per-instance state // thay vì global state }

3.2 Zustand Store thay đổi

// === THAY THẾ === isBuilding: boolean; verticesBuild: IVerticesBuild | null; flowBuildStatus: Record<string, ...>; flowPool: Record<string, ...>; // === BẰNG === buildInstances: Map<string, BuildInstance>; // Helper selectors isAnyBuilding: () => boolean; // Trả về true nếu bất kỳ build instance nào có status 'running' getBuildingInstancesForNode(nodeId: string): BuildInstance[]; // Trả về tất cả build instances có chứa nodeId trong verticesToRun getBuildStatusForNode(nodeId: string): BuildStatus; // Merge trạng thái từ tất cả build instances có chứa node đó // Nếu bất kỳ instance nào đang BUILDING node → BUILDING

Phase 2: Build Flow Refactor

- Cập nhật buildInstances[buildId].verticesBuild thay vì get().verticesBuild

- Cập nhật buildInstances[buildId].flowPool thay vì get().flowPool

- queryClient.invalidateQueries giữ nguyên (có thể global vì là cache invalidation)
- Gọi processBuildQueue() để chạy build pending kế tiếp

8. Risks & Mitigations

RiskImpactMitigation
HIGH Shared flowPool trong component nào đó được đọc bởi cả 2 build cùng lúc Data race — output sai Flow pool cần per-build instance hoặc dùng runId làm key
MED Edge animation bị conflict (build 1 build B, build 2 build D → cùng set edge animation) Visual glitch Merge status: running > to_build > built. Edge animation ưu tiên running
MED Component re-render loop khi nhiều build instances cùng update state Performance degradation Batch updates với React 18 auto-batching. Dùng useSyncExternalStore cho performance-critical paths
LOW Backend resource contention (CPU, RAM) khi nhiều build chạy song song Chậm hơn nếu tài nguyên cạn User đã xác nhận backend có thể xử lý. Monitor sau deploy
LOW set({ flowBuildStatus: {}, latestRunningText: {} }) ở đầu buildFlow reset state của build khác Lost state Đã xử lý trong Phase 2: chuyển reset từ global → per-build-instance
