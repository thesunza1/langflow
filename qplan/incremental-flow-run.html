Phân tích hiện trạng (GitNexus)

Backend — Đã hỗ trợ sẵn

src/backend/base/langflow/api/build.py đã có:

  • stop_component_id: str | None — parameter để dừng build tại component cụ thể
  • start_component_id: str | None — parameter để bắt đầu từ component cụ thể
  • graph.sort_vertices(stop_component_id, start_component_id) — sort vertices theo start/stop
  • graph.stop_vertex — nếu có, chỉ chạy tới component đó rồi dừng

Kết luận: Backend engine đã sẵn sàng cho tính năng "run tới C là dừng".

Frontend — Cần cải tiến

src/frontend/src/utils/buildUtils.ts đã có:

  • stopNodeId — truyền vào API build
  • startNodeId — truyền vào API build

Vấn đề: UI chưa có cơ chế để user chọn "chạy từ đầu" hay "reuse cache" khi thêm component mới.

Functional Requirements

FR1: Run tới component được chọn là dừng

Mô tả: Khi user bấm "Run" trên 1 component C trong flow A→B→C→D, chỉ chạy A→B→C, không chạy D.

Cách thực hiện: Frontend truyền stop_component_id = id(C) vào API build. Backend sort_vertices(stop_component_id, ...) đã hỗ trợ sẵn.

File ảnh hưởng:

  • src/frontend/src/stores/flowStore.ts — update buildFlow action để truyền stopNodeId
  • src/frontend/src/utils/buildUtils.ts — đảm bảo stopNodeId được dùng đúng
  • src/frontend/src/components/core/flowToolbarComponent/components/playground-button.tsx — run button handler

CẢI TIẾN

FR2: Popup chọn "Run từ đầu" vs "Reuse từ node gần nhất"

Mô tả: Khi thêm component mới (E) sau D đã chạy, bấm Run trên E → popup hỏi:

  • "Run từ component gần nhất" (mặc định) → chạy từ D→E, dùng cache A,B,C,D
  • "Run từ đầu" → chạy lại toàn bộ A→B→C→D→E

Cách thực hiện: Frontend cần:

  • Phát hiện upstream nodes đã có kết quả (dựa vào flowBuildStatus hoặc buildInfo)
  • Hiển thị popup với 2 tuỳ chọn khi bấm Run trên 1 node đã có cache upstream
  • Nếu chọn "reuse": gọi API với start_component_id = node đã chạy gần nhất
  • Nếu chọn "run từ đầu": gọi API bình thường (không có start_component_id)

File cần tạo mới:

  • src/frontend/src/modals/runModeModal/index.tsx — popup component

File cần sửa:

  • src/frontend/src/stores/flowStore.ts — thêm logic check cache và xử lý tuỳ chọn
  • src/frontend/src/utils/buildUtils.ts — updateVerticesOrder hỗ trợ start_component_id

MỚI

FR3: Xác định upstream node gần nhất đã chạy

Mô tả: Khi user bấm Run trên node E, hệ thống cần xác định node upstream gần nhất đã có cached result để đề xuất reuse.

Logic:

  • Xác định các direct predecessors của node được run (từ edge list)
  • Kiểm tra từng predecessor đã có buildStatus === BuildStatus.BUILT chưa
  • Predecessor gần nhất có result → đó là điểm bắt đầu cho reuse

MỚI

Non-functional Requirements

  • NFR1: Popup chỉ xuất hiện khi có upstream node đã build thành công (không popup nếu chưa có cache nào)
  • NFR2: Mặc định chọn "Run từ component gần nhất" (an toàn, nhanh hơn)
  • NFR3: Không làm thay đổi behavior của "Run All" (playground button)
  • NFR4: Cache check dựa trên flowBuildStatus trong flowStore

Implementation Plan

  1. FR1 (Run & dừng): Sửa buildFlow trong flowStore.ts — khi run 1 component, truyền stopNodeId là id của component đó vào API.
  2. FR2 (Popup): Tạo runModeModal component với 2 lựa chọn. Hook vào sự kiện run component.
  3. FR3 (Cache detection): Thêm helper function findNearestBuiltUpstream trong buildUtils.ts để xác định upstream node gần nhất đã chạy.
  4. FR2 tiếp: Kết nối popup với API call — nếu chọn "reuse" thì dùng start_component_id.

Scope

In Scope:
  • Backend API đã hỗ trợ stop_component_id/start_component_id — không cần sửa backend
  • Sửa frontend flowStore.buildFlow để dùng stopNodeId
  • Tạo popup component mới (runModeModal)
  • Thêm helper tìm upstream node gần nhất đã build
  • Xử lý run từ component gần nhất (dùng start_component_id)
Out of Scope:
  • Không sửa backend graph engine
  • Không thay đổi playground/chat behavior
  • Không thêm cơ chế persist cache (dùng cache service hiện tại)
  • Không thêm node-level caching indicator trong graph view
